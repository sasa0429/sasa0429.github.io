版本控制

版本控制是指对软件开发过程中各种程序代码、说明文档等文件的变更进行管理。

它将追踪文件变化，记录文件的变更时间、变更内容、甚至变更执行人进行记录，同时对每一个阶段性变更（不仅仅只是一个文件的变化）添加版本编号，方便将来进行查阅特定阶段的变更信息，甚至是回滚。

人工版本控制器

通过人工的复制行为来保存项目的不同阶段的内容，添加适当的一些描述文字加以区分。

缺点：

  • 繁琐、容易出错
  • 产生大量重复（冗余）数据

版本控制工具

通过程序完成上述人工版本控制行为。

优点：

  • 方便且功能强大
  • 只记录不同版本之间变化的部分

扩展：工作流 - git work flow

master：一般功能不在master上开发，否则很容易造成冲突，而且不易维护，master分支是对外的分支。

现在要开发一项目：

  • 首先创建仓库
  • 然后在master分支上迁出一开发分支，这个分支作为主开发分支，
  • 在这个主开发分支上，可能因为有其他的功能模块要开发，根据不同的功能迁出不同的feature子分支，比如：用户分支，产品分支…
  • 在开发完成一个分支之后，会合并到主开发分支。
  • 开发后期阶段可以测试上线成一个版本了，此时并不是直接merge到master上，首先放到Release分支上，一般主要用来做测试，如果在这个过程中测试通过了，直接合并到master上，如果在这个过程中出现了bug修复，会在Release分支上开其他的分支进行修复，修复完再合并到当前release分支上，再进行测试，直到通过，合并到master上。在master中如果觉得当前可以成为一个版本了，可以打上tag标签处理。
  • 剩下的开发分支继续进行。
  • 如果在主分支在开发过程中形成了很多个版本，如果某天出现了某个bug，比如在v0.1有bug，这时可单独从master分支中迁出一个专门用来修复的Hotfix分支，修复完在合并回master，合并完之后形成v0.2。

git 忽略文件

例如：webstorm生成的.idea配置文件很多时候不需要与其他人共享，不是项目中所必要的，没必要要把它纳入到版本控制中去，但如果不作处理，每次提交Git都会提示这个文件夹的未被追踪。这时可以在当前目录下新建git忽略文件夹，然后把当前目录或者文件写到忽略文件列表中，此时Git就会忽略对这些文件的追踪了
